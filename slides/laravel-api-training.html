Day 3

API Security

Students add authentication, token abilities, frontend token validation, and rate limiting to the route group.

PDF pages 11-136 hours
Day 3Frontend token plus throttling plus expiring Sanctum token plus abilities
Security pipeline

Every protected request passes through layered checks.

Client-> X-API-TOKEN?-> throttle:60,1-> auth:sanctum-> abilities-> Controller-> JSON

Missing frontend token

Return 401 with a clear JSON error.

Too many requests

Laravel returns 429 to protect the API.

Token denied

Sanctum returns 401 for missing or expired tokens and 403 for missing abilities.

Security ruleOne layer is never the whole security system
Security route group

Route grouping keeps security rules obvious.

Public route

Route::post('/auth/login', [AuthController::class, 'login'])
    ->middleware(['frontend.token', 'throttle:login']);

Protected routes

Route::middleware([
    'frontend.token',
    'auth:sanctum',
    'throttle:api',
])->group(function () {
    Route::apiResource('users', UserProfileController::class)
        ->middlewareFor(['index', 'show'], 'abilities:profiles:read')
        ->middlewareFor('store', 'abilities:profiles:create')
        ->middlewareFor('update', 'abilities:profiles:update')
        ->middlewareFor('destroy', 'abilities:profiles:delete');
});
Day 3 detailLogin can be public to users, but not open to unknown frontend clients

Token abilities

The `abilities` column turns one bearer token into scoped access.

Stored in MySQL

SELECT id, name, abilities, expires_at
FROM personal_access_tokens
ORDER BY id DESC
LIMIT 1;
["profiles:read","profiles:create",
"profiles:update","profiles:delete"]

Day 3 - React auth

Protected browser CRUD calls need both API security headers.

Login response

{
  "message": "Login successful.",
  "data": {
    "token_type": "Bearer",
    "access_token": "1|plain-text-token",
    "expires_at": "2026-06-09T09:30:00Z",
    "abilities": [
      "profiles:read",
      "profiles:create",
      "profiles:update",
      "profiles:delete"
    ]
  }
}

Protected request headers

{
  'X-API-TOKEN': FRONTEND_API_TOKEN,
  Authorization: `Bearer ${token}`
}
Security modelFrontend token identifies the client; bearer token identifies the user; abilities control allowed actions

Performance And Exception Handling

Students improve query behavior, cache API responses, and standardize JSON errors without weakening Day 3 security.

PDF pages 14-186 hours
Day 4Pagination plus eager loading plus cache plus exceptions
Performance model

Optimize reads, invalidate on writes.

GET users->ability read->cache key->cache hit?
cache miss->UserProfile::with('projects')->store cache
POST/PATCH/DELETE->write ability->write model->clear cache

Key practices

  • paginate every list endpoint
  • use eager loading to avoid N+1 queries
  • include page/search filters in cache keys
  • clear stale cache after writes
PerformanceFast APIs are deliberate about payloads, queries, cache, and frontend UX
